# Compat fixtures

## Overview

hermes-station ships two test fixtures under `tests/fixtures/`:

| Fixture | Status | Purpose |
|---|---|---|
| `data-fresh/` | Committed (generated by conftest) | Verifies the contract for a brand-new first-boot deploy |
| `data-realistic/` | **Gitignored** | Verifies the contract against real-world long-running volume state |

The realistic fixture is intentionally gitignored — it is generated from a live Railway `/data` snapshot and may contain personal configuration even after sanitization. This document explains how to generate it.

## Why it exists

The synthetic `data-fresh/` fixture only covers the "just deployed, nothing configured yet" state. Real deployments accumulate state over time: populated `config.yaml`, pairing tokens, channel configs, `memories/`, and so on. `test_compat_realistic.py` exercises the config reader, path resolution, and admin status endpoint against that real-world shape to catch regressions that the fresh fixture cannot surface.

The test file auto-skips when `tests/fixtures/data-realistic/.hermes/` does not exist, so the default `uv run pytest` run is unaffected by its absence.

## How to generate the fixture

### Step 1 — take a snapshot of the live Railway volume

The Railway CLI does not support direct file download — `railway run` executes a
command inside the service container but provides no way to retrieve files it
creates. To pull a snapshot from a live Railway deployment, use the **Railway
dashboard volume browser** (Project → your service → Volumes → Browse) to
download the `/data` directory, or use `railway shell` to get an interactive
session and pipe the tarball to stdout:

```bash
# Opens an interactive shell in the running hermes-station service on Railway
railway shell --service hermes-station

# Inside the Railway shell, stream the tarball to a local file via SSH-style redirection:
# (run this from the railway shell, piping to a local terminal session)
tar -czf - -C / data > /tmp/data.tgz
# Then exit the shell and retrieve /tmp/data.tgz via the volume browser,
# or use `railway shell` with a here-document to pipe to stdout and redirect locally.
```

> **Practical tip:** The simplest approach for a Railway-hosted deployment is to
> use the Railway dashboard's volume browser to download the `/data` volume
> contents, then unpack locally and pass the result to the sanitize script.

Alternatively, if you are running hermes-station locally:

```bash
# Apple container
container exec hermes-station tar -czf /tmp/data.tgz -C / data
container cp hermes-station:/tmp/data.tgz ./data.tgz

# Docker
docker exec hermes-station tar -czf /tmp/data.tgz -C / data
docker cp hermes-station:/tmp/data.tgz ./data.tgz
```

### Step 2 — run the sanitize script

```bash
./scripts/sanitize-data-snapshot.sh <path-to-snapshot.tgz>
```

This writes the sanitized fixture to `tests/fixtures/data-realistic/` by default. Pass a second argument to override the destination:

```bash
./scripts/sanitize-data-snapshot.sh data.tgz tests/fixtures/data-realistic
```

The script performs eight steps automatically:

1. Extracts the tarball to a staging area, auto-detecting whether the archive root is `/data`, `./data`, or bare contents.
2. Copies only the three directories governed by `docs/CONTRACT.md`: `.hermes/`, `webui/`, `workspace/`. Everything else (`.cache/`, `.npm/`, `.local/`, `lost+found/`, user content) is discarded.
3. **Scrubs secrets**: replaces all values in `.hermes/.env` with `PLACEHOLDER_<KEY>` (keys are preserved so config parsing still works), and replaces all values in `.hermes/pairing/*.json`, `.hermes/platforms/pairing/*.json`, and `.hermes/auth.json` with `"PLACEHOLDER"`.
4. **Deletes** PII directories: `sessions/`, `memories/`, `logs/`, `sandboxes/`, `bin/`, `cache/`, `cron/` (each directory is kept but emptied). `workspace/` is emptied entirely.
5. **Deletes** SQLite databases and their WAL/SHM sidecars (`state.db`, `memory_store.db`, `kanban.db`) and regenerable runtime state (`gateway.lock`, `gateway.pid`, `gateway_state.json`, `processes.json`). Also deletes the `.hermes/gateway/` subdirectory (per-platform gateway state, regenerated on boot).
6. Replaces `webui/.signing_key` with 32 random bytes (the test only checks that the file persists, not its value). Clears `webui/sessions/`.
7. Writes a `MANIFEST.txt` to the fixture root recording what was scrubbed and when.
8. Prints the sanitized fixture size on disk.

### Step 3 — manual review

The script preserves a few files that are non-sensitive by design but that you should still review before sharing the fixture:

- `.hermes/config.yaml` — provider name and model slug. Non-secret but may reveal your setup.
- `.hermes/SOUL.md` — your agent's personality prompt. May contain personal preferences.
- `.hermes/skills/` and `optional-skills/` — skill catalog. Usually generic, but custom skills may contain personal content.
- `.hermes/channel_directory.json`, `models_dev_cache.json` — cached metadata.

The `MANIFEST.txt` in the fixture root lists exactly what was scrubbed and what was preserved.

## Expected directory structure after sanitization

```
tests/fixtures/data-realistic/
  MANIFEST.txt
  .hermes/
    config.yaml           # kept; non-secret config
    .env                  # kept; values replaced with PLACEHOLDER_<KEY>
    SOUL.md               # kept; review manually
    channel_directory.json
    models_dev_cache.json
    pairing/              # any .json files have values scrubbed
    platforms/pairing/    # same
    skills/               # kept
    optional-skills/      # kept (if present)
    sessions/             # emptied
    memories/             # emptied
    logs/                 # emptied
    sandboxes/            # emptied
    bin/                  # emptied
    cache/                # emptied
    cron/                 # emptied
    # state.db, memory_store.db, kanban.db — DELETED
    # gateway.lock, gateway.pid, gateway_state.json, processes.json — DELETED
    # gateway/ — DELETED (per-platform runtime state; regenerated on boot)
  webui/
    .signing_key          # replaced with 32 random bytes
    sessions/             # emptied
  workspace/              # emptied (dir kept)
```

The minimum required for `test_compat_realistic.py` to detect and activate the suite is:

```
tests/fixtures/data-realistic/.hermes/   (directory must exist)
```

And for `test_fixture_has_expected_skeleton` to pass:

```
tests/fixtures/data-realistic/webui/.signing_key
```

## Running the realistic compat tests

Once the fixture is populated:

```bash
uv run pytest tests/test_compat_realistic.py -v
```

Expected: all tests pass. If `test_sanitized_env_loads_cleanly` fails, the `.env` was not sanitized correctly — re-run the sanitize script.
